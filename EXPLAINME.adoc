= Burble — Show Me The Receipts
:toc:
:icons: font

The README makes claims. This file backs them up.

If you're reading this, you probably saw "26,350x speedup" or "sub-microsecond
PTP" and thought "yeah, right." Fair. Here's how each headline feature actually
works, with file paths so you can verify.

== "26,350x LZ4 Speedup" — How?

Not comparing apples to oranges. Same operation, same data, different execution
path.

**Elixir path:** `Enum.reduce/3` over a byte list, pattern matching each byte,
building a compressed binary via `<<>>` concatenation. Elixir is a functional
language on the BEAM VM — it's brilliant for concurrency, terrible for byte-level
loops. 83ms for an audio frame.

**Zig path:** SIMD-vectorised LZ4 running as a NIF (Native Implemented Function)
loaded into the BEAM via `erlang:load_nif/2`. Zig compiles to native machine code
with auto-vectorisation. The NIF boundary adds ~200ns overhead. 3.1 microseconds
total.

**The code:**
- Zig implementation: `ffi/zig/src/main.zig` (search for `lz4_compress`)
- NIF interface: `server/lib/burble/coprocessor.ex`
- Benchmark: `server/lib/mix/tasks/bench_coprocessor.ex` — run it yourself with
  `cd server && mix bench.coprocessor`

The speedup is real because we're comparing an interpreted byte loop against
compiled SIMD. Same reason you wouldn't write a video codec in Python.

== "Sub-Microsecond PTP" — Not Marketing Nonsense

IEEE 1588 Precision Time Protocol is a real standard used in financial trading,
broadcast, and industrial control. We didn't invent it — we integrated it.

**What we actually do:**

1. Probe the kernel for `/dev/ptp*` hardware clock devices (these exist on NICs
   with PTP support — Intel i210, Mellanox, etc.)
2. If no hardware PTP, check for `phc2sys` (a daemon that bridges PTP clocks to
   the system clock)
3. If no phc2sys, fall back to NTP (which gets you ~1ms, not microseconds)
4. If nothing, use the system clock and warn

**The code:** `server/lib/burble/timing/ptp.ex` — 594 lines, fully documented.

The GenServer samples clock offset every 5 seconds, maintains a 60-sample sliding
window, and exports standard deviation as "jitter" via Erlang telemetry. The media
plane uses this to compensate for clock drift between nodes when mixing audio
streams.

**The honest caveat:** You only get sub-microsecond if your NIC has a PTP hardware
clock _and_ your network has a PTP grandmaster. On a regular cloud VM, you'll get
NTP-quality timing (~1ms). The code handles this gracefully — it doesn't lie about
what it has.

== "Fault Isolation Per Room" — OTP Isn't Magic, But It's Close

When we say "if a room crashes, everything else keeps running," this isn't a
framework feature we added — it's how Erlang/OTP works by design. It was built
for telephone switches in the 1980s that needed 99.9999999% uptime.

**How it works in Burble:**

Each room is a `GenServer` process started under a `DynamicSupervisor`. If a room
process crashes (bad packet, codec error, out of memory), the supervisor:

1. Detects the crash via process monitoring
2. Logs the failure
3. Optionally restarts the room with a clean state
4. Does NOT affect any other room, the auth system, presence, or signaling

This is not a try/catch. Processes in Erlang are lightweight (2KB each), isolated
(separate heaps), and communicate only via message passing. A crash in one process
physically cannot corrupt another process's memory.

**The code:** `server/lib/burble/application.ex` — look at the supervision tree.
Each `Burble.Room` is a child of `Burble.RoomSupervisor` (a DynamicSupervisor).

**The honest caveat:** If VeriSimDB goes down, all rooms lose their backing store.
OTP isolates process failures, not infrastructure failures. We mitigate this with
connection pooling and backoff retries in `Burble.Store`.

== "Self-Hostable" — Actually One Command

Not "one command after you install Docker, Node, PostgreSQL, Redis, and a reverse
proxy." Actually one command:

[source,bash]
----
cd containers && podman-compose -f compose.toml up
----

This starts three Chainguard-based containers:
1. `burble-server` — Elixir release with embedded Zig NIFs
2. `burble-web` — Static ReScript client served by Caddy
3. `verisimdb` — Database (our own, not PostgreSQL)

Total image size: ~120MB. Compare to Jitsi (~2GB) or Matrix Synapse (~500MB).

**Why Chainguard:** Minimal attack surface. No shell in production images, no
package manager, signed with Sigstore cosign, daily security updates.

**The code:** `containers/compose.toml`, `containers/Containerfile.server`

== "Mumble Bridge" — Real Bidirectional, Not One-Way

The Mumble bridge maintains a persistent TCP connection to a Murmur server
speaking Mumble's custom protobuf-over-TCP protocol on port 64738.

**What actually flows:**
- Voice packets (Opus codec) are relayed bidirectionally
- User presence (join/leave/channel move) is synced
- Text messages are forwarded
- Channel mapping is configurable (Mumble channel → Burble room)

**What doesn't flow:**
- Mumble's permission system is separate from Burble's
- File transfers aren't bridged
- Mumble's positional audio isn't translated (yet)

**The code:** `server/lib/burble/bridges/mumble.ex`

== "SIP Bridge" — Yes, Real SIP

`server/lib/burble/bridges/sip.ex` implements a subset of RFC 3261. SIP is a
text-based protocol (think HTTP-for-phones). The bridge:

1. Listens for SIP INVITE/BYE/REGISTER on UDP and TCP
2. Maps SIP sessions to Burble rooms
3. Transcodes G.711/G.722 ↔ Opus via the Zig coprocessor
4. Handles NAT with STUN/TURN

This means Burble rooms can receive calls from desk phones, PBXes, and SIP
trunks. A conference room phone can join the same voice channel as browser
users.

**The honest caveat:** SIP is a nightmare protocol. Our implementation covers
the common call flows but doesn't handle every edge case in the 269-page RFC.
Registration, basic call setup/teardown, and DTMF work. Complex call transfer
scenarios may not.

**The code:** 1,400+ lines in `server/lib/burble/bridges/sip.ex`
