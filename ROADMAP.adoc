= Burble Roadmap
:toc: preamble

== Philosophy: "Majestic Mumble"
At its core, Burble is designed to be **as easy to use and simple for starters as Mumble**. A user should be able to switch directly over with zero configuration, enjoying a lightweight, low-latency, and frictionless voice experience. 

However, beneath this simple surface lies a **"Majestic"** architecture. For power users, organizations, and high-security environments, Burble provides a "Dual-Track" system featuring formally verified proofs, zero-trust network perimeters (SDP), predictive routing (AWOL), and post-quantum cryptography readiness.

== Current Status

Burble has cleared its foundational milestones. Test suite: 222+ Elixir tests + Zig coprocessor tests + 6 channel safety-contract tests + 6 Opus-contract tests + 5 AI bridge Deno tests (round-trip, ordering, reconnect, heartbeat). We are currently in a **phased hardening** cycle: audio reliability (Phase 1), P2P AI bridge dependability (Phase 2), then signaling/channels (Phase 3) and PTP hardware integration (Phase 4). See `.machine_readable/6a2/STATE.a2ml` for the authoritative status.

== Milestones

=== v0.1.0 to v0.4.0 — Foundation & Transport (DONE)

* [x] WebRTC voice engine (SFU, Opus, ExWebRTC)
* [x] E2EE via Insertable Streams (X25519 + AES-256-GCM)
* [x] VeriSimDB integration & LMDB zero-copy playout buffering
* [x] QUIC transport (0-RTT, multiplexed streams)
* [x] Zig SIMD coprocessor (audio, DSP, neural, compression)
* [x] Groove discovery endpoint (/.well-known/groove)

=== v1.0.0 — Stable Release (DONE)

* [x] QUIC hardening (rate limiting, datagram size enforcement)
* [x] Bebop codegen & wire protocol schemas
* [x] Idris2 ABI formal proofs (Avow consent, Vext integrity, Permissions)
* [x] Chainguard base Containerfile pinned by digest
* [x] **Rate Limiting:** ETS Token Bucket algorithm for authentication paths.

=== v1.1.0 — High Rigor & Resilience (Current Phase - Partially Sorted)

* [x] **SDP Foundation:** Software-Defined Perimeter zero-trust logic with Zig FFI `nftables` bridge.
* [x] **AWOL Routing:** Layline predictive routing (RTT/loss velocity) for seamless handover.
* [ ] **LLM Service:** QUIC + TCP transport scaffold exists; **provider not wired, `parse_frame` has a bug, worker pool is no-op** (Phase 2b — secondary to P2P AI bridge).
* [x] **P2P AI Bridge:** Claude-to-Claude JSON data channel over WebRTC. Both send and receive legs working (receive-leg bug fixed 2026-04-16). Heartbeat, status UI, round-trip + ordering + reconnect tests. This is the **critical path** for pair-programming.
* [x] **Groove Protocol:** Health Mesh inter-service probing and Feedback routing.
* [x] **Formal Proofs:** `MediaPipeline.idr` (linear buffer consumption) and `WebRTCSignaling.idr` (JSEP transitions) implemented. **Avow attestation is data-type-only — no dependent-type enforcement at runtime yet** (Phase 1 target: hash-chain audit log).
* [x] **Accessibility Scaffolding:** Server-side screen reader generation and persistent keyboard keybinding management.
* [ ] **Circuit Breakers & Health Checks:** Prevent cascading failures and ensure auto-recovery.
* [ ] **Automated Backups:** Disaster recovery for VeriSimDB state.
* [x] **Idris2 Compilation:** Fully compile remaining proofs to C headers for Zig ABI validation.
* [x] **File Send Executable Isolation:** Implement `chmod` lockdown for file transfers.
* [x] **Internationalisation:** Link Zig FFI API to world corpus language support (`standards/lol`). _(V-lang API layer removed 2026-04-16; Zig FFI is now the sole surface.)_

== The "Majestic" Backlog (To-Do)

Based on recent comprehensive audits, the following areas must be addressed before reaching enterprise-grade scale and compliance.

=== 1. Accessibility (Target: WCAG 2.3 AAA, minimum AA)
* [ ] **Screen Reader Support:** Full testing with NVDA/VoiceOver and client-side ARIA focus traps.
* [ ] **Sign Language & Avatars:** Implement `SignLanguage.ex` with interpreting queues.
* [ ] **Braille Support:** Complete `braille.ex` integration for specialized hardware.
* [ ] **Alternative Pointers:** Eye-tracking and touch optimizations for motor-impaired users.
* [ ] **Contrast & Reflow:** Audit UI for strict contrast ratios and responsive reflowing without breaking.

=== 2. Dependability & Platform
* [ ] **Chaos Engineering:** Implement resilience testing (network partitioning, node failures).
* [ ] **Auto-Scaling:** Kubernetes Horizontal Pod Autoscaler (HPA) integration.
* [ ] **Multi-Server Redundancy:** Expand from single-server supervision to distributed clustering.
* [ ] **Automated Updates:** Zero-downtime rolling deployments via CI/CD.

=== 3. Performance & Caching
* [ ] **Multi-Level Cache:** Add ETS cache for frequent reads, Redis for distributed sessions, and edge caching (CDN) for static assets.
* [ ] **WebRTC Tuning:** Adaptive bandwidth and dynamic jitter buffer sizing.
* [ ] **Audio Batching:** Further optimize Zig NIFs with batched frame processing to minimize BEAM context switches.
* [ ] **Database Tuning:** Covering indexes and query analysis for VeriSimDB.

=== 4. Security & Hardening
* [ ] **Blockchain Anchoring:** Tamper-proof logs and provenance auditing.
* [ ] **Automated Secret Rotation:** Key management lifecycle for E2EE keys.
* [ ] **Quantum-Resistant Crypto:** Prepare hybrid post-quantum cipher suites.
* [ ] **Full SDP Rollout:** Enforce Single Packet Authorisation (SPA) on all edge nodes.

=== 5. Developer Experience & Tooling
* [ ] **Must-Have Tooling Rollout:** Enforce Credo, Dialyzer, and GitHub Actions in the primary pipeline.
* [ ] **Should-Have Integrations:** Add `Proven` for formal verification, `Bypass` for testing, and `Telemetry` dashboards.
* [ ] **Aspirational Features:** Livebook interaction, WebAssembly (Wasm) client builds, and EXLA GPU acceleration.
* [ ] **Documentation:** Expand examples and onboarding guides for the Zig FFI coprocessor API.

=== 6. User UX/UI
* [ ] **Mobile Touch Optimization:** Ensure large touch targets and simplified gestures.
* [ ] **Onboarding Polish:** Guided tooltips mimicking the simplicity of legacy VoIP clients.
* [ ] **Error Handling:** Granular, user-friendly error messages for WebRTC or SDP rejections.
